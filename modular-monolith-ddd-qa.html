Builder's Inner Monologue

Building a Modular Monolith
with DDD

The real questions a developer asks themselves — before writing a single line of code.

35 Questions · 9 Sections
01 The Big Picture
Q.01 Why am I building a Modular Monolith and not microservices? +

Microservices solve scale and team autonomy problems — but introduce enormous operational complexity: distributed tracing, network failures, eventual consistency everywhere, and independent deployments. If you don't have those problems yet, you're paying the cost without the benefit.

A Modular Monolith gives you clean internal boundaries while keeping deployment simple. You can always extract a module into a microservice later — you can't easily untangle a messy monolith.

Start with a Modular Monolith. Migrate a module to a microservice only when a specific, measurable problem forces you to.
Q.02 What are my modules? How do I identify them? +

Modules map to Bounded Contexts in DDD — self-contained slices of the business where a specific model and language applies. Ask yourself:

  • What are the core business capabilities? (Orders, Catalog, Customers, Payments)
  • Where does the meaning of a word change? "Customer" in Billing ≠ "Customer" in Shipping
  • What could change independently without affecting everything else?

Don't over-split — a module should represent a meaningful, cohesive business capability, not a technical layer.

Q.03 What's the one rule I absolutely cannot break between modules? +

Modules must never call each other directly. No method calls, no shared repositories, no shared database tables. The only allowed cross-module communication:

  • Asynchronous events via an Events Bus (RabbitMQ, Kafka, etc.)
  • Public integration event contracts — only the IntegrationEvents assembly of another module, never its internals
If you allow direct calls, your "modules" are just folders. You'll have a distributed monolith in disguise — all the coupling, none of the clarity.
02 Module Internals
Q.04 How do I structure the inside of a module? +

Each module follows Clean Architecture split into four assemblies:

AssemblyResponsibility
DomainBusiness rules — Entities, Aggregates, Value Objects, Domain Events
ApplicationUse cases — Commands, Queries, Event Handlers, Validators
InfrastructureDB access, Event Bus, Background jobs, Module initialization
IntegrationEventsPublic event contracts other modules can subscribe to

Dependency direction: Infrastructure → Application → Domain. Domain has zero external dependencies.

Q.05 How does the API talk to a module? Do I expose the whole module? +

No — the API only sees a thin interface per module with exactly three methods:

public interface IOrdersModule
{
    Task<TResult> ExecuteCommandAsync<TResult>(ICommand<TResult> command);
    Task ExecuteCommandAsync(ICommand command);
    Task<TResult> ExecuteQueryAsync<TResult>(IQuery<TResult> query);
}

The API calls ExecuteCommandAsync(new CreateOrderCommand(...)) and has no idea what runs inside. This is the module's entire public surface.

Q.06 Each module has its own IoC container — why not just one global one? +

A single DI container makes all services from all modules visible to each other — it's trivially easy to inject something you shouldn't, breaking module isolation silently.

Each module's own Composition Root enforces that its internal services are invisible to other modules, each module manages its own lifecycle, and initialization is explicit via Initialize(...) called by API startup.

03 Domain Model
Q.07 What goes in the Domain layer? What doesn't? +

In Domain: Aggregate Roots, Entities, Value Objects, Domain Events, Domain Services, Business Rules.

Not in Domain: anything touching infrastructure — no EF Core, no HTTP clients, no loggers, no repositories. The domain is pure C# POCOs with zero external dependencies. This is Persistence Ignorance.

A good test: can you run the entire Domain layer in a unit test with zero mocking of external dependencies? If yes, you've done it right.
Q.08 What is an Aggregate Root and how do I design one? +

An Aggregate Root is the single gatekeeper for a cluster of related objects. You never modify child objects directly:

// ❌ BAD — bypasses invariants
order.Items.Add(new OrderItem(productId, quantity));

// ✅ GOOD — root enforces rules
order.AddItem(productId, quantity);
// Inside: validate business rules, raise domain event

The root enforces business invariants, raises Domain Events, and controls access to children. Design rule: load the whole aggregate at once, save the whole aggregate at once.

Q.09 I keep using Guid and string everywhere. Is there a better way? +

Use Value Objects to avoid Primitive Obsession:

public record OrderId(Guid Value)
{
    public static OrderId New() => new(Guid.NewGuid());
}

// Compile-time safety — can't accidentally pass wrong ID type:
Order order = _repo.GetById(new CustomerId(id)); // ❌ type mismatch

Apply to all domain concepts: Money, Email, Address. Catches entire classes of bugs at compile time and makes code self-documenting.

Q.10 Two users edit the same record at the same time — who wins? +

Without protection the last write silently overwrites the first — a "lost update." The solution is Optimistic Concurrency via Entity Version:

User A reads Order (Version = 1)
User B reads Order (Version = 1)

User A saves → WHERE Id=1 AND Version=1 → success → Version=2 ✅
User B saves → WHERE Id=1 AND Version=1 → 0 rows affected ❌ → conflict

User B gets an exception, re-fetches the updated data, and retries. No silent data loss. No database row locks needed.

04 CQRS
Q.11 Why separate Commands from Queries? Isn't it more code for the same result? +

Yes, more code — but reads and writes have fundamentally different needs:

Commands (Write)Queries (Read)
ModelRich domain model (DDD)Flat DTOs, raw SQL
FocusBusiness rules, validationProjections only
Side effectsYes — changes stateNever — read only

Without CQRS, your query handler loads a full Aggregate just to return { orderId, status, total }. Wasteful and fragile. With CQRS, a query runs one targeted SQL query and returns a DTO.

Q.12 How does a Command actually get processed? What's the pipeline? +

The Command flows through a decorator chain before reaching your handler:

API Controller
  → module.ExecuteCommandAsync(new CreateOrderCommand(...))
      → LoggingDecorator        // logs start, args, errors
          → ValidationDecorator    // runs FluentValidation
              → UnitOfWorkDecorator    // wraps in DB transaction
                  → CreateOrderHandler // your business logic
              // UoW commits + dispatches domain events

Your handler knows nothing about logging, validation, or transactions. Pure Single Responsibility.

05 Events
Q.13 Domain Event vs Integration Event — when do I use each? +
  • Domain Event = "something happened, handle it within this module" — synchronous, in-process, same transaction
  • Integration Event = "something happened, tell the outside world" — asynchronous, via message bus, separate transaction
CreateOrderCommand
  → Order raises OrderCreatedDomainEvent
      → updates inventory (same transaction)
      → schedules OrderCreatedIntegrationEvent
          → published to Event Bus (after commit)
              → Email module, Shipping module react
A Domain Event never leaves the module. An Integration Event is the module's public announcement to the world.
Q.14 What if the app crashes after saving to the DB but before publishing the event? +

Classic dual-write problem — the order is saved but the event is never sent. Silent data loss. The solution is the Outbox Pattern:

  • Save the integration event to OutboxMessages table in the same transaction as the business operation
  • A background job reads unprocessed messages and publishes them to the Event Bus
  • Once published, the message is marked processed

If the app crashes, the Outbox message survives. On restart, the background job retries. At-least-once delivery, guaranteed.

Q.15 What if I receive the same integration event twice? +

The Inbox Pattern solves this:

  • When an event arrives, check InboxMessages for its unique event ID
  • If already processed → skip it silently
  • If new → process it and record it in the Inbox

Outbox (producer) + Inbox (consumer) = exactly-once processing semantics even on an unreliable message bus.

06 Advanced Patterns
Q.16 I need to know the full history of an entity, not just its current state. How? +

Event Sourcing — store every event that ever happened and derive current state by replaying them:

// Traditional DB: you only see the result
Order { Status: "Shipped", Total: 350 }

// Event Sourcing: you see everything
[OrderCreated, ItemAdded, DiscountApplied, PaymentConfirmed, Shipped]
// Replay → derive current state
Only use Event Sourcing where auditability and full history are genuine business requirements — financial transactions, legal records, etc.
Q.17 Some tasks are slow — I don't want the user to wait. How do I handle async work inside a module? +

Use Internal Commands — scheduled in the same transaction as the originating command, but processed asynchronously by a background job without crossing module boundaries:

// During CreateOrder handling:
_commandScheduler.Schedule(
    new SendOrderConfirmationInternalCommand(order.Id)
);
// Saved to DB in same transaction. Background job picks it up.

User gets a fast response. Slow work (PDF generation, emails, third-party APIs) happens in the background. The UnitOfWorkDecorator marks them processed to prevent duplicate execution.

Q.18 Catalog has a "Product." Orders also has a "Product." Should they share a model? +

No. Each module has a completely different view:

  • Catalog Product: name, description, images, categories, SEO metadata
  • Orders Product: product ID, price at time of order, quantity — that's it

Sharing a model couples two modules to the same change cycle. Orders stores a snapshot of the product at order time and subscribes to ProductPriceChangedIntegrationEvent if it needs updates.

Duplication in data models is acceptable and expected in DDD. Coupling is the real enemy.
Q.19 How do I test this properly? Where do I even start? +

Test in three layers:

  • Domain unit tests — pure C#, no mocks, no DB. Verify business rules fire, invariants hold, domain events are raised. Fast and numerous.
  • Application integration tests — real DB via test containers, run a full command through the pipeline, assert resulting state and published events.
  • Module contract tests — verify integration events your module publishes match what consuming modules expect. Catches breaking changes early.
Because Domain has no infrastructure dependencies, unit tests require zero setup. That's the reward for Persistence Ignorance.
Q.20 When is a Modular Monolith no longer the right choice? +

Consider extracting a module into a microservice when you have specific, measurable pressure:

  • A module needs to scale independently (high-traffic search module)
  • A module needs a completely different tech stack (ML engine in Python)
  • Different teams own different modules and deployment coupling causes real friction
  • A module has dramatically different availability or latency SLAs

Because you enforced clear boundaries, data isolation, and async-only communication from the start — extraction is a matter of moving code and rewiring Event Bus connections externally.

A well-built Modular Monolith is a microservices architecture waiting to be extracted — not a legacy system to escape.
07 Data & Persistence
Q.21 Each module has its own schema — but it's still one database, right? Is that a problem? +

One database, separate schemas per module — and that's a valid starting point. The goal is logical isolation, not necessarily physical. Each module only reads and writes its own schema. No cross-schema joins. Benefits:

  • Simpler ops — one DB to backup, monitor, and manage
  • No distributed transactions needed yet
  • Each module's schema can evolve independently

If a module later needs its own database for scaling, you can move it cleanly because no other module was ever touching its tables.

Shared schema = modules that can secretly bypass each other's boundaries via SQL. Separate schemas make that impossible at the DB level.
Q.22 How do I handle database migrations? One project or one per module? +

One migration project per module. Each module owns its schema entirely — including how it evolves over time:

// Module initialization runs its own migrations
public static void Initialize(...)
{
    ConfigureCompositionRoot(...);
    DatabaseMigrator.Migrate(connectionString, schema: "orders");
    QuartzStartup.Initialize(...);
}

Teams can ship schema changes to their module without coordinating with other teams. A single shared migration project is a hidden coupling mechanism — avoid it.

Q.23 How do I map my domain objects to the database without leaking infrastructure into the domain? +

Keep all mapping configuration in the Infrastructure layer using EF Core's IEntityTypeConfiguration:

public class OrderEntityTypeConfiguration
    : IEntityTypeConfiguration<Order>
{
    public void Configure(EntityTypeBuilder<Order> builder)
    {
        builder.ToTable("Orders", schema: "orders");
        builder.HasKey(o => o.Id);
        builder.OwnsOne(o => o.TotalPrice);  // maps Value Object
        builder.HasMany(o => o.Items).WithOne();
    }
}

The Order domain class has no EF attributes — no [Column], no [Key]. It's a pure POCO. The mapping is 100% in Infrastructure.

Q.24 Should I use the Repository pattern or just inject DbContext directly? +

Repository for writes (Commands). DbContext directly for reads (Queries).

For Commands: a repository abstracts persistence, keeps the Application layer independent of EF Core, and makes domain logic testable:

// Application layer — no EF dependency
public interface IOrdersRepository
{
    Task<Order> GetByIdAsync(OrderId id);
    void Add(Order order);
}

For Queries: inject a read-only connection directly into the query handler and run a targeted SQL query. No domain model involved — no need for the abstraction overhead.

Repository = aggregates you load, modify, and save. Query handlers = raw data access for projections.
08 Error Handling & Resilience
Q.25 How do I handle business rule violations? Do I throw exceptions from the domain? +

Yes — but use a dedicated domain exception type, not a generic Exception:

public void Cancel()
{
    if (Status == OrderStatus.Shipped)
        throw new BusinessRuleValidationException(
            "Cannot cancel an order that has already been shipped."
        );

    Status = OrderStatus.Cancelled;
    AddDomainEvent(new OrderCancelledDomainEvent(Id));
}

The API's global exception middleware maps BusinessRuleValidationException to 400 Bad Request, and infrastructure exceptions to 500. The mapping logic lives entirely outside the domain.

Q.26 A downstream module's event handler fails — does the whole system break? +

No — and this is a key benefit of async communication. If the Email module's handler fails, the order is still saved. The Event Bus retries delivery (configurable with exponential backoff). After max retries, the message goes to a Dead Letter Queue (DLQ) for manual inspection.

Because your Inbox Pattern makes handlers idempotent, retries are safe — processing the same event twice produces the same result.

Design every integration event handler to be idempotent. If the handler runs twice, it must produce the exact same outcome both times.
Q.27 How do I roll back changes if something fails mid-command? +

The UnitOfWorkDecorator wraps the entire handler in a database transaction. If anything throws before Commit(), the transaction rolls back automatically:

UnitOfWorkDecorator
  → begins transaction
  → handler runs (modifies aggregates)
  → domain events dispatched
  → if all OK: UnitOfWork.Commit()  // persists everything atomically
  → if exception: transaction rolls back  // nothing saved

Never call SaveChanges() directly inside a handler. Let the decorator own the commit boundary. All or nothing.

Q.28 How do I handle validation errors before they even reach the domain? +

The ValidationDecorator runs before the handler using FluentValidation — checking input shape and basic rules:

public class CreateOrderCommandValidator
    : AbstractValidator<CreateOrderCommand>
{
    public CreateOrderCommandValidator()
    {
        RuleFor(x => x.CustomerId).NotEmpty();
        RuleFor(x => x.Items).NotEmpty()
            .WithMessage("Order must contain at least one item.");
        RuleForEach(x => x.Items)
            .ChildRules(i => i.RuleFor(x => x.Quantity).GreaterThan(0));
    }
}

If validation fails, a ValidationException is thrown immediately — the domain never executes, no DB round-trip wasted. The validator is the bouncer; the domain is the business inside.

09 Team & Real-World Concerns
Q.29 My team uses the word "Order" to mean different things in different contexts. How do I handle this? +

This is exactly what Ubiquitous Language solves. The same word can — and should — mean different things in different Bounded Contexts:

  • Orders context: items, total, status, shipping address
  • Finance context: a payment record with invoice details
  • Warehouse context: a pick-list of items to pull from shelves

Each module has its own Order class. They share only an ID, passed via integration events. This is correct and intentional — don't try to unify them.

When two teams argue about what "Order" means, that's a signal they're in different Bounded Contexts. Split the word, not the team.
Q.30 How do I add a new feature that spans multiple modules? +

You don't build a single cross-module feature — you build collaborating reactions. Each module does its part in response to events.

Example: "When a premium order is placed, upgrade loyalty tier and notify the warehouse."

Orders module:
  CreateOrderCommand → PremiumOrderPlacedIntegrationEvent published

Customers module:
  Subscribes → upgrades loyalty tier internally

Warehouse module:
  Subscribes → creates priority pick list

No module knows what the others are doing. To add a new reaction (Notifications module), you just subscribe to the existing event — zero changes to existing code. Open/Closed Principle at the architecture level.

Q.31 How do I trace a request across modules for debugging? +

Use Correlation IDs. Every incoming HTTP request gets a unique ID that is:

  • Stored in the ExecutionContext — the current-user/request context passed to each module
  • Written to every log entry by the LoggingDecorator
  • Attached to every integration event as a property
  • Passed through to any Internal Commands spawned during the request

To trace a flow: filter your log aggregator (Seq, Kibana, Datadog) by correlation ID. You see the full chain across all modules — from the original HTTP request to every downstream event handler — as one linear trace.

Q.32 How do I handle authorization? Does each module check permissions separately? +

Authorization at two levels:

  • API level (coarse-grained) — controllers check authentication and role before forwarding commands. Standard ASP.NET Core [Authorize] attributes handle this.
  • Domain level (fine-grained) — business rules encode ownership and permission logic: "only the order's owner can cancel it," "a warehouse manager can override stock."

The user's identity is passed into each module via IExecutionContextAccessor — initialized at the API level and injected into handlers. The domain uses it to enforce ownership rules with zero dependency on HTTP or authentication infrastructure.

Q.33 How do I handle a long-running process that spans multiple steps and modules? +

Use a Saga (Process Manager) — a stateful coordinator that listens to events and issues commands to keep a long-running flow moving forward:

OrderFulfillmentSaga:

1. Hears OrderCreatedIntegrationEvent
   → sends ReserveStockCommand to Warehouse

2. Hears StockReservedIntegrationEvent
   → sends ProcessPaymentCommand to Payments

3. Hears PaymentProcessedIntegrationEvent
   → sends ScheduleShipmentCommand to Shipping

// On failure → compensating commands (e.g., release reserved stock)

The Saga holds state of where the process is. If any step fails, it issues compensating actions. This replaces distributed transactions with choreography through events.

Q.34 What lives in BuildingBlocks and what's the risk of it becoming a dumping ground? +

BuildingBlocks should contain only reusable infrastructure abstractions with no business domain specificity:

  • Base classes: Entity, AggregateRoot, ValueObject
  • CQRS interfaces: ICommand, IQuery, ICommandHandler
  • Event Bus abstractions, Outbox/Inbox infrastructure, UnitOfWork, shared exceptions

Never put there: business logic, domain models, or module-specific types. The moment you add an OrderStatus enum to BuildingBlocks, every module is now coupled to Orders.

BuildingBlocks is the "toolkit shelf." If a module puts its domain objects there for convenience, challenge it immediately — it's a coupling smell.
Q.35 How do I know if my module boundaries are correct? What are the warning signs they're wrong? +

Warning signs your boundaries are wrong:

  • Chatty event storms — every small action in Module A fires an event that Module B immediately reacts to. They should be one module.
  • Shared aggregates — two modules both need to modify the same entity. Ownership is unclear. Extract a new module or merge them.
  • Integration events carry domain objects — your event payload looks like a rich domain model rather than a flat DTO. You're coupling consumers to your internals.
  • Teams coordinate schema changes constantly — if two teams can't release independently, their modules are too tightly coupled.
  • A feature requires changing 4 modules at once — the boundary is drawn in the wrong place.
Good module boundaries let a single team make most changes entirely within their module. If cross-module changes are routine, restructure.
